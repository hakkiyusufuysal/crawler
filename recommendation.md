# Production Deployment Recommendations

To move this crawler into a high-scale production environment, the first priority is replacing the single-process SQLite storage with a distributed data layer. The inverted index should be migrated to Elasticsearch or Apache Solr for sub-second full-text search over billions of documents, while the crawl frontier and visited-URL set should be backed by Redis (using Redis Sets for O(1) deduplication and Sorted Sets for priority-based URL scheduling). The crawler workers should be deployed as stateless containers (Kubernetes pods) pulling URLs from a shared message queue (Kafka or RabbitMQ), enabling horizontal scaling across dozens of machines. Per-domain rate limiting should be centralized via a Redis-backed token bucket so that all workers collectively respect each domain's capacity. DNS resolution should be cached aggressively, and a connection pool with HTTP/2 multiplexing (via a library like `httpx`) would dramatically reduce per-request overhead at scale.

For operational reliability, the system needs a proper observability stack: Prometheus metrics for crawl throughput, error rates, queue depth, and per-domain latency, visualized through Grafana dashboards with alerting thresholds. The crawler should implement politeness policies beyond robots.txt — including crawl-delay headers, exponential backoff on 429/503 responses, and configurable domain-level concurrency caps. Content deduplication via SimHash or MinHash would prevent indexing near-duplicate pages (common on large sites). For fault tolerance, each URL fetch should be an idempotent operation with at-least-once delivery semantics from the queue, and the system should checkpoint its frontier state to durable storage every N seconds so that any worker crash loses at most a few seconds of progress. Finally, a scheduling layer (Airflow or a cron-based controller) would manage recurring crawls, re-indexing stale content, and garbage-collecting expired pages from the index.
